# 관리자/CMS 설계안

## 목표

졸업 후 다른 관리자가 코딩을 몰라도 웹사이트에 로그인해서 내용을 수정할 수 있어야 합니다.

관리자 화면은 “대시보드 + 콘텐츠 에디터” 형태가 적합합니다.

## 관리자에서 수정 가능해야 할 항목

- 홈페이지 메인 문구
- 연구소 소개
- 비전/미션/핵심가치
- 핵심과제
- 연구실 소개
- 연구진/자문위원/운영위원
- 연구성과
- 공지사항/자료실/뉴스
- 연락처/주소/지도
- 첨부파일/PDF
- 공개/비공개 상태

## 추천 DB 테이블

```text
site_settings
pages
page_sections
labs
people
people_roles
research_outputs
posts
comments
attachments
inquiries
admin_audit_logs
```

## 테이블 역할

`site_settings`

- 대표 연구소명
- 대표 전화
- 대표 이메일
- 주소
- SNS/외부 링크
- SEO 기본값

`pages`

- 페이지 단위 정보
- 예: 홈페이지, 소개, 연구성과, 연락처

`page_sections`

- 페이지 안의 섹션 단위 콘텐츠
- 예: 비전 섹션, 미션 섹션, 핵심가치 섹션

`labs`

- 연구실 정보
- 이름, 설명, 색상, 정렬 순서, 공개 여부

`people`

- 사람 정보
- 이름, 소속, 직위, 연구분야, 이메일, 전화, 사진, 공개 여부

`people_roles`

- 사람이 어떤 역할인지 연결
- 예: 연구소장, 실장, 연구위원, 객원연구위원, 자문위원

`research_outputs`

- 연구성과
- 제목, 요약, 카테고리, 연도, 저자, 파일, 링크, 공개 여부

`posts`

- 공지사항/자료실/뉴스 게시글

`inquiries`

- 문의 폼 제출 내역
- 처리 상태, 담당자 메모 포함

`admin_audit_logs`

- 누가 언제 무엇을 바꿨는지 기록
- 운영 안정성에 중요

## 관리자 UX

관리자 화면은 다음 메뉴가 좋습니다.

- 대시보드
- 페이지 편집
- 연구진 관리
- 연구성과 관리
- 게시판 관리
- 문의 관리
- 사이트 설정
- 사용자/권한 관리

## 에디터 방식

초기에는 복잡한 블록 에디터보다 안정적인 폼 기반 편집을 권장합니다.

예:

- 제목 input
- 요약 textarea
- 본문 rich text 또는 markdown editor
- 이미지 업로드
- 공개 여부 toggle
- 정렬 순서 number input
- 저장/미리보기 버튼

## 권한 모델

최소 권한은 다음처럼 나눕니다.

- `admin`: 모든 관리 가능
- `editor`: 콘텐츠 작성/수정 가능
- `viewer`: 관리자 화면 읽기만 가능

원본의 `user_roles`는 `admin`, `user` 두 단계뿐이라 리뉴얼에서 확장하는 것이 좋습니다.

## 중요한 보안 원칙

- 관리자 API는 서버에서 권한을 다시 검사해야 합니다.
- service role key는 서버에만 보관해야 합니다.
- 파일 업로드는 허용 확장자와 크기 제한을 둬야 합니다.
- 공개 개인정보는 필드별 공개 여부를 둬야 합니다.
- 삭제는 즉시 영구 삭제보다 `archived` 또는 `deleted_at` 방식을 우선 고려합니다.
