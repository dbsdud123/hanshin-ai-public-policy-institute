# 초보자용 개념 설명

## 웹사이트는 크게 무엇으로 구성되나요?

웹사이트는 보통 세 덩어리로 이해하시면 됩니다.

1. 화면
2. 데이터 저장소
3. 배포 서버

원본 프로젝트에서는 다음처럼 대응됩니다.

- 화면: Next.js, React, Tailwind CSS
- 데이터 저장소: Supabase Database, Supabase Storage
- 로그인: Supabase Auth
- 배포 서버: Vercel

## Next.js란?

Next.js는 React로 웹사이트를 만들기 위한 프레임워크입니다.

React가 화면 조각을 만드는 도구라면, Next.js는 페이지 이동, 서버 렌더링, 이미지 최적화, 배포 최적화까지 같이 도와주는 틀입니다.

원본에서는 `app/` 폴더 안의 파일들이 실제 페이지입니다.

예:

- `app/page.tsx`: 홈페이지
- `app/about/page.tsx`: 소개 페이지
- `app/members/page.tsx`: 연구진 페이지

## Supabase란?

Supabase는 웹사이트에 필요한 서버 기능을 대신 제공하는 서비스입니다.

원본에서는 다음 기능을 씁니다.

- 회원가입/로그인
- 게시글 저장
- 댓글 저장
- 첨부파일 저장
- 관리자 역할 저장

## DB란?

DB는 데이터를 표처럼 저장하는 공간입니다.

예를 들어 게시글 DB는 대략 이런 표입니다.

```text
id | title | content | author_name | created_at
```

관리자가 공지사항을 쓰면 이 표에 한 줄이 추가되고, 웹사이트는 그 줄을 읽어서 화면에 보여줍니다.

## CMS란?

CMS는 Content Management System의 약자입니다.

쉽게 말하면 “코딩 없이 웹사이트 내용을 고치는 관리자 화면”입니다.

이번 프로젝트의 핵심은 단순히 예쁜 사이트를 만드는 것이 아니라, 연구소 소개글이나 연구진 정보를 관리자가 직접 고칠 수 있는 CMS를 만드는 것입니다.

## Vercel이란?

Vercel은 만든 웹사이트를 인터넷에 공개해 주는 서비스입니다.

개발자는 코드를 GitHub에 올리고, Vercel은 그 코드를 빌드해서 주소를 만들어 줍니다.

현재 원본 사이트의 주소가 `vercel.app`으로 끝나는 이유도 Vercel에 배포되어 있기 때문입니다.

## 환경변수란?

환경변수는 코드에 직접 넣으면 안 되는 설정값입니다.

예:

- Supabase 주소
- Supabase 공개 키
- 관리자 서버용 비밀 키

중요한 비밀값은 GitHub에 올리면 안 되고, Vercel이나 로컬 `.env.local`에만 보관해야 합니다.

## RLS란?

RLS는 Row Level Security의 약자입니다.

DB 안에서 “누가 어떤 줄을 읽고/쓰고/수정할 수 있는지” 정하는 보안 규칙입니다.

예:

- 누구나 공지사항을 읽을 수 있음
- 로그인한 사람만 댓글 작성 가능
- 작성자 본인만 자기 글 수정 가능
- 관리자는 모든 글 수정 가능

Supabase를 공식 웹사이트 운영용으로 쓰려면 RLS를 매우 꼼꼼히 설계해야 합니다.
